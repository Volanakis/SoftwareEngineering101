# Cinema Management Backend System

Backend διαχείρισης κινηματογραφικών προγραμμάτων (σεζόν) και προβολών, υλοποιημένο ως σύνολο RESTful web services. Η εργασία εκπονείται στο πλαίσιο του μαθήματος **321-4002 – Software Engineering** (παράδοση: 14/01/2026), βάσει του εγγράφου εκφώνησης `ΕΡΓΑΣΙΑ ΜΑΘΗΜΑΤΟΣ 2025 - ΟΜΑΔΕΣ ΕΝΟΣ Ή ΔΥΟ ΑΤΟΜΩΝ.pdf`.

Το σύστημα καλύπτει όλο τον κύκλο ζωής ενός προγράμματος — από τη δημιουργία του, την υποβολή προβολών, την ανάθεση προσωπικού, τις αξιολογήσεις και τον προγραμματισμό, μέχρι την τελική δημόσια ανακοίνωση. Δεν περιλαμβάνει πώληση εισιτηρίων ή εγγραφή θεατών.

## Τεχνολογική Στοίβα

Η υλοποίηση (Τρίτο Μέρος Εργασίας) θα γίνει σε **Python** με:

| Ανάγκη (κατά το εκφώνημα) | Επιλογή |
|---|---|
| Γλώσσα υλοποίησης | Python 3 |
| RESTful framework | **Flask** (Blueprints ανά πόρο: Program, Screening) |
| Πρόσβαση σε δεδομένα / ORM | **SQLAlchemy** |
| Διαχείριση εξαρτήσεων & πακέτο εργασιών | **Poetry** (αντίστοιχο του Maven για Python) |
| Unit testing | **pytest** (αντίστοιχο του JUnit) |
| Rate limiting | **Flask-Limiter** |
| Βάση δεδομένων | Σχεσιακή βάση (π.χ. PostgreSQL/SQLite), κοινή με το User Management System για τον πίνακα Users |
| Έλεγχος εκδόσεων | Git + ιδιωτικό αποθετήριο GitHub |

Αυτές είναι οι Python-ισοδύναμες επιλογές που προτείνει ρητά το εκφώνημα για όποιον δεν χρησιμοποιήσει Java/Maven/JUnit/Jersey.

## Λειτουργικές Απαιτήσεις

Αριθμημένες ιεραρχικά (`ΛΑ-<ενότητα>.<α/α>`). Πηγή: ενότητες FUNCTIONS/REQUIREMENTS, ROLES, ENTITIES του εγγράφου εκφώνησης.

### ΛΑ-1: Χρήστες, Αυθεντικοποίηση & Εξουσιοδότηση

- **ΛΑ-1.1** Κάθε χρήστης διαθέτει μοναδικό username, password (hash) και πλήρες όνομα (fullName).
- **ΛΑ-1.2** Το σύστημα αυθεντικοποιεί τον χρήστη ελέγχοντας τον συνδυασμό username/password έναντι της κοινής βάσης δεδομένων (η ίδια βάση εξυπηρετεί και το εξωτερικό User Management System).
- **ΛΑ-1.3** Ένας μη αυθεντικοποιημένος χρήστης αντιμετωπίζεται ως **VISITOR**.
- **ΛΑ-1.4** Ένας αυθεντικοποιημένος χρήστης χωρίς ρόλο σε συγκεκριμένο πρόγραμμα έχει τα ίδια δικαιώματα αναζήτησης/προβολής με τον VISITOR, επιπλέον της δυνατότητας δημιουργίας προγράμματος/προβολής (ρόλος **USER**).
- **ΛΑ-1.5** Ένας χρήστης μπορεί να έχει το πολύ έναν ρόλο (**PROGRAMMER** ή **STAFF**) ανά συγκεκριμένο πρόγραμμα, αλλά διαφορετικούς ρόλους σε διαφορετικά προγράμματα.
- **ΛΑ-1.6** Ο ρόλος **SUBMITTER** δεν είναι εγγεγραμμένος ρόλος του προγράμματος (PROGRAMMERS/STAFF) — προκύπτει έμμεσα από τη δημιουργία μιας προβολής (ο δημιουργός γίνεται αυτόματα SUBMITTER αυτής).
- **ΛΑ-1.7** Ένας PROGRAMMER ενός προγράμματος δεν επιτρέπεται να υποβάλει προβολές (SUBMITTER) στο ίδιο πρόγραμμα, ώστε να διασφαλίζεται η αμεροληψία.
- **ΛΑ-1.8** Κάθε αίτημα προς οποιαδήποτε λειτουργία διαχείρισης ελέγχεται ως προς την εξουσιοδότηση του αιτούντος με βάση τον ρόλο του στο συγκεκριμένο πρόγραμμα/προβολή· μη εξουσιοδοτημένα αιτήματα απορρίπτονται με κατάλληλο μήνυμα σφάλματος.

### ΛΑ-2: Διαχείριση Προγράμματος (Season)

- **ΛΑ-2.1 Δημιουργία προγράμματος**
  - **ΛΑ-2.1.1** Επιτρέπεται μόνο αν το όνομα (name) είναι μοναδικό και όλα τα υποχρεωτικά πεδία (name, description, startDate, endDate) έχουν τιμή.
  - **ΛΑ-2.1.2** Το σύστημα παράγει αυτόματα το id και το creationDate.
  - **ΛΑ-2.1.3** Η αρχική κατάσταση είναι πάντα `CREATED`.
  - **ΛΑ-2.1.4** Ο δημιουργός (τρέχων χρήστης) καταχωρείται αυτόματα ως PROGRAMMER του προγράμματος.
- **ΛΑ-2.2 Ενημέρωση προγράμματος**
  - **ΛΑ-2.2.1** Επιτρέπεται μόνο σε PROGRAMMER του συγκεκριμένου προγράμματος.
  - **ΛΑ-2.2.2** Επιτρέπεται τροποποίηση name, description, startDate/endDate και των συνόλων PROGRAMMERS/STAFF.
  - **ΛΑ-2.2.3** Ο δημιουργός δεν μπορεί ποτέ να αφαιρεθεί από το σύνολο PROGRAMMERS.
  - **ΛΑ-2.2.4** Δεν επιτρέπονται ενημερώσεις μετά την είσοδο του προγράμματος στην κατάσταση `ANNOUNCED`.
- **ΛΑ-2.3 Προσθήκη PROGRAMMER**: επιτρέπεται σε PROGRAMMER, εφόσον ο προστιθέμενος χρήστης δεν είναι ήδη PROGRAMMER του προγράμματος (αλλιώς επιστρέφεται σφάλμα).
- **ΛΑ-2.4 Προσθήκη STAFF**: επιτρέπεται σε PROGRAMMER· μετά την είσοδο σε κατάσταση `SUBMISSION`, το σύνολο STAFF παγώνει (καμία περαιτέρω μεταβολή).
- **ΛΑ-2.5 Αναζήτηση προγραμμάτων**: με προαιρετικά κριτήρια name, description, dates, filmTitle, auditorium, συνδυαζόμενα με λογικό **AND**· χωρίς κριτήρια επιστρέφονται όλα· τα αποτελέσματα φιλτράρονται ανά ρόλο του αιτούντος και ταξινομούνται πρώτα κατά ημερομηνία και μετά κατά όνομα.
- **ΛΑ-2.6 Προβολή προγράμματος**: παρουσίαση στοιχείων βάσει id, με τα εμφανιζόμενα πεδία redacted ανάλογα με τον ρόλο του αιτούντος (π.χ. ο Visitor βλέπει μόνο δημόσια στοιχεία).
- **ΛΑ-2.7 Διαγραφή προγράμματος**: επιτρέπεται σε οποιονδήποτε PROGRAMMER του προγράμματος, μόνο εφόσον αυτό βρίσκεται σε κατάσταση `CREATED`.
- **ΛΑ-2.8 Μετάβαση κατάστασης προγράμματος**: επιτρέπεται μόνο σε PROGRAMMER· δεν επιτρέπεται επιστροφή σε προηγούμενη κατάσταση (rollback) ούτε παράλειψη ενδιάμεσου βήματος. Οι επιτρεπτές μεταβάσεις είναι αποκλειστικά:
  - **ΛΑ-2.8.1** `CREATED → SUBMISSION` (έναρξη υποβολών)
  - **ΛΑ-2.8.2** `SUBMISSION → ASSIGNMENT` (κλείσιμο υποβολών, έναρξη ανάθεσης STAFF)
  - **ΛΑ-2.8.3** `ASSIGNMENT → REVIEW` (έναρξη αξιολογήσεων)
  - **ΛΑ-2.8.4** `REVIEW → SCHEDULING` (μόνο έγκριση/απόρριψη προβολών πλέον)
  - **ΛΑ-2.8.5** `SCHEDULING → FINAL_SUBMISSION` (τελικές υποβολές αλλαγών)
  - **ΛΑ-2.8.6** `FINAL_SUBMISSION → DECISION` (τελικές αποφάσεις· κάθε εγκεκριμένη προβολή που δεν έχει τελικά υποβληθεί απορρίπτεται αυτόματα)
  - **ΛΑ-2.8.7** `DECISION → ANNOUNCED` (κλείδωμα περιεχομένου, δημόσια ανακοίνωση)

### ΛΑ-3: Διαχείριση Προβολών (Screening)

- **ΛΑ-3.1 Δημιουργία προβολής**: μόνο από USER· υποχρεωτική σύνδεση με συγκεκριμένο πρόγραμμα· ο δημιουργός γίνεται αυτόματα SUBMITTER· αρχική κατάσταση `CREATED`.
- **ΛΑ-3.2 Ενημέρωση προβολής**: επιτρέπεται στον SUBMITTER μόνο όσο state=`CREATED`· επεξεργάσιμα πεδία: auditorium, film, startTime, endTime.
- **ΛΑ-3.3 Υποβολή προβολής**: `CREATED → SUBMITTED`· επιτρέπεται μόνο στον SUBMITTER, μόνο αν το πρόγραμμα είναι σε `SUBMISSION` και η προβολή είναι πλήρης (film, auditorium, παραγόμενο endTime από τη διάρκεια της ταινίας).
- **ΛΑ-3.4 Απόσυρση προβολής**: επιτρέπεται στον SUBMITTER μόνο όσο state=`CREATED`· η προβολή διαγράφεται οριστικά (δεν έχει εισέλθει σε επίσημη αξιολόγηση).
- **ΛΑ-3.5 Ανάθεση χειριστή (STAFF)**: κατά τη φάση `ASSIGNMENT` του προγράμματος, ο PROGRAMMER αναθέτει ακριβώς ένα μέλος STAFF ως handler.
- **ΛΑ-3.6 Αξιολόγηση προβολής**: ο ανατεθειμένος STAFF καταχωρεί αριθμητική βαθμολογία και αναλυτικά σχόλια· επιτρέπεται μόνο όσο το πρόγραμμα είναι σε `REVIEW`· `→ REVIEWED`.
- **ΛΑ-3.7 Έγκριση προβολής**: επιτρέπεται μόνο σε PROGRAMMER, μόνο όσο το πρόγραμμα είναι σε `SCHEDULING`· `→ APPROVED`· μπορεί να συνοδεύεται από σημειώσεις για τελικές αλλαγές.
- **ΛΑ-3.8 Απόρριψη προβολής**: `→ REJECTED` (τελική κατάσταση). Δύο μηχανισμοί:
  - **ΛΑ-3.8.1** Χειροκίνητη, από PROGRAMMER: στο `SCHEDULING` (βάσει αξιολόγησης) ή στο `DECISION` (αν η τελική υποβολή δεν καλύπτει τις απαιτούμενες αλλαγές).
  - **ΛΑ-3.8.2** Αυτόματη, από το σύστημα: στο `DECISION`, κάθε εγκεκριμένη αλλά μη τελικά υποβληθείσα προβολή.
  - **ΛΑ-3.8.3** Σε κάθε περίπτωση καταγράφεται υποχρεωτικά η αιτιολογία απόρριψης.
- **ΛΑ-3.9 Τελική υποβολή προβολής**: επιτρέπεται στον SUBMITTER, μόνο όσο το πρόγραμμα είναι σε `FINAL_SUBMISSION`· μετά την επιτυχή εκτέλεση τα στοιχεία «παγώνουν» (freeze).
- **ΛΑ-3.10 Αποδοχή προβολής (τελικός προγραμματισμός)**: επιτρέπεται μόνο σε PROGRAMMER, μόνο όσο το πρόγραμμα είναι σε `DECISION`, για εγκεκριμένη & τελικά υποβληθείσα προβολή· `→ SCHEDULED` (τελική κατάσταση).
- **ΛΑ-3.11 Αναζήτηση προβολών**: εντός συγκεκριμένου προγράμματος, με προαιρετικά κριτήρια filmTitle, cast, genre, εύρος ημερομηνιών· σημασιολογία **AND** μεταξύ κριτηρίων· εντός πεδίου κειμένου, όλες οι λέξεις πρέπει να εμφανίζονται (case-insensitive)· ταξινόμηση κατά genre και μετά τίτλο ταινίας, ή κατά start_time για προβολή χρονοδιαγράμματος.
- **ΛΑ-3.12 Προβολή στοιχείων προβολής**: παρουσίαση βάσει id, με redaction πεδίων ανάλογα με τον ρόλο του αιτούντος (owner/handler/programmer/visitor).

## Μη Λειτουργικές Απαιτήσεις

Αριθμημένες ιεραρχικά (`ΜΛΑ-<α/α>`). Πηγή: παράγραφος αξιοπιστίας/ασφάλειας/απόδοσης του εκφωνήματος.

- **ΜΛΑ-1 Απόδοση**: οι περισσότερες λειτουργίες του συστήματος πρέπει να εκτελούνται εντός 5–10 δευτερολέπτων ανά αίτημα.
- **ΜΛΑ-2 Αξιοπιστία**
  - **ΜΛΑ-2.1** Το σύστημα δεν πρέπει να παρουσιάζει εσωτερικά σφάλματα (5xx) λόγω λανθασμένης χρήσης — πρέπει να επιστρέφει κατάλληλα, κατανοητά μηνύματα σφάλματος σε περίπτωση εσφαλμένης, μη έγκυρης ή ελλιπούς εισόδου.
  - **ΜΛΑ-2.2** Πρέπει να αποτρέπεται η πολλαπλή εκτέλεση μιας ήδη επιτυχημένης, μη ιδεμποτεντικής ενέργειας (π.χ. διπλή δημιουργία προγράμματος/προβολής), ενώ ιδεμποτεντικές ενέργειες (π.χ. προβολή στοιχείων) μπορούν να επαναληφθούν χωρίς παρενέργειες.
- **ΜΛΑ-3 Ασφάλεια**
  - **ΜΛΑ-3.1** Όλοι οι χρήστες πρέπει να αυθεντικοποιούνται και να αντιστοιχίζονται σε ρόλους.
  - **ΜΛΑ-3.2** Η πρόσβαση σε κάθε λειτουργία υπόκειται σε κανόνες ελέγχου πρόσβασης (role-based access control) σύμφωνα με τον πίνακα ρόλων.
  - **ΜΛΑ-3.3** Εφαρμόζεται περιορισμός ρυθμού αιτημάτων (rate limiting), ιδίως σε ενέργειες όπως υποβολή προβολών και αναζητήσεις, για αποτροπή κατάχρησης.
- **ΜΛΑ-4 Συναλλακτικότητα**: κάθε ενημέρωση στη βάση δεδομένων πρέπει να είναι συνεπής (consistent) και να ολοκληρώνεται επιτυχώς μόνο εφόσον η αντίστοιχη λειτουργία (π.χ. προγραμματισμός προβολής) έχει εκτελεστεί πλήρως και ορθά (all-or-nothing transactions).
- **ΜΛΑ-5 Καταγραφή & Ιχνηλασιμότητα**: το σύστημα πρέπει να διαθέτει μηχανισμούς σωστού χειρισμού σφαλμάτων (error handling) και καταγραφής (logging) που να διατηρούν ιστορικό (audit trail) όλων των σημαντικών ενεργειών, για διευκόλυνση της αντιμετώπισης προβλημάτων.
- **ΜΛΑ-6 Επεκτασιμότητα/Συντηρησιμότητα**: η αρχιτεκτονική οργανώνεται σε σαφείς στρώσεις (presentation/service/data access, βλ. component diagram) ώστε να διευκολύνεται η συντήρηση και η μελλοντική επέκταση.
- **ΜΛΑ-7 Τεχνολογικοί περιορισμοί**: η συνεργατική υλοποίηση πρέπει να πραγματοποιείται μέσω Git/ιδιωτικού GitHub repository, με χρήση εργαλείου διαχείρισης εξαρτήσεων/build (Poetry) και πλαισίου unit testing (pytest), σύμφωνα με τις απαιτήσεις του Τρίτου Μέρους της εργασίας.
